# TypeScript Web Resources

## Project structure

```text
src/WebResources/
├── Lib/      # shared utilities, no direct Xrm event bindings
├── Form/     # form scripts, one module per form/entity
├── View/     # view-related scripts
├── Ribbon/   # command bar enable/display rule and command handlers
├── package.json
└── tsconfig.json
```

This mirrors the previous version of this guideline — keep the four-folder split rather than
flattening everything into one folder, even on a project with relatively few scripts. It keeps
form scripts, shared library code, and command bar handlers from blurring together as a project
grows.

## Register `OnLoad` only — bind everything else in code

Register only the form's **`OnLoad`** event from the Dataverse form designer. All other events
(`OnSave`, `OnChange` for specific fields, tab/section visibility handlers, etc.) are
registered programmatically from inside your `OnLoad` handler, in code:

```typescript title="Form/account.ts"
export function onLoad(executionContext: Xrm.Events.EventContext): void {
  const formContext = executionContext.getFormContext();

  formContext.data.entity.addOnSave(onSave);
  formContext.getAttribute("name")?.addOnChange(onNameChange);
}

function onSave(executionContext: Xrm.Events.SaveEventContext): void {
  // ...
}

function onNameChange(executionContext: Xrm.Events.EventContext): void {
  // ...
}
```

This keeps the full set of event bindings for a form visible in one place in source control,
rather than split between the form's XML and the script — a form's event wiring shouldn't
require opening the form designer to understand.

## Naming and the early-bound model

Use the [generated TypeScript model](model-generation.md) for form/entity typing rather than
hand-typed `Xrm.Attributes` lookups by string — it gives you the same compile-time safety on
the client that early-bound `.cs` classes give you on the server, and is generated by the same
`dgtp codegeneration` command.

## Linting and formatting

- Visual Studio Code default formatting.
- Enforce `@microsoft/eslint-plugin-power-apps` lint rules in CI — a build that passes locally
  but fails lint in CI is a sign the local editor isn't picking up the shared config; fix the
  editor setup, don't relax the CI rule.

## `.gitattributes`

Enforce one line-ending convention for TypeScript sources the same way as for C# — see
[Source Control](../../alm/source-control.md#gitattributes). Mixed line endings between
contributors on different OSes is the most common source of noisy diffs in client-side
projects specifically, since editors here vary more than on the .NET side.
